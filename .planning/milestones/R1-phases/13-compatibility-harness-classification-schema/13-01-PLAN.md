Resolve the latest stable version of each of the six plugins as of today (the harness build date). For each slug, query the wp.org API:

```
curl -s "https://api.wordpress.org/plugins/info/1.2/?action=plugin_information&request[slug]=SLUG&request[fields][version]=1"
```

Slugs: `woocommerce`, `jetpack`, `wordpress-seo` (Yoast SEO), `elementor`, `wpforms-lite` (WPForms free / WPForms Lite), `lifterlms`. Extract each `version` field. (Use Yoast SEO, NOT Rank Math — locked decision; load `wordpress-seo` only.)

Confirm each versioned ZIP URL is real before committing it:
`https://downloads.wordpress.org/plugin/SLUG.VERSION.zip` — a `curl -sI` should return 200, not 404. If a 404, fall back to the unversioned `https://downloads.wordpress.org/plugin/SLUG.zip` only as a last resort and note it in VERSIONS.md; prefer an exact version.

Write `tests/compat/.wp-env.json` in a single pass, mirroring the root shape but:
- `core`: `"WordPress/WordPress#7.0"` (match root env — locked)
- `plugins`: array of the six versioned `downloads.wordpress.org/plugin/SLUG.VERSION.zip` URLs
- `mappings`: `{ "wp-content/plugins/maestro-menu-editor": "../.." }` (Maestro from source, relative to tests/compat/)
- `port`: `8890`, `testsPort`: `8891` (distinct from root 8888/8889 so both envs run side-by-side)
- `config`: `{ "WP_DEBUG": true, "SCRIPT_DEBUG": true }`
- `lifecycleScripts.afterStart`: the user-provisioning command described below

**afterStart user provisioning (admin + Editor + Shop Manager).** afterStart runs AFTER wp-env installs and activates the plugins, so WooCommerce (and thus the `shop_manager` role) should be registered by then — but guard it explicitly so the run never half-fails. Follow the get-or-create idempotent pattern from tests/e2e/global-setup.ts (try `wp user get`, create only if missing) so re-running `wp-env start` is safe. afterStart runs inside the cli container, so call `wp ...` directly (not `wp-env run`). The command must:
1. Get-or-create an Editor: if `wp user get compat_editor` fails, `wp user create compat_editor compat-editor@example.com --role=editor --user_pass=password`.
2. Guard Shop Manager on the role existing — use a literal `wp role exists shop_manager` (or `wp role list --field=name | grep -qx shop_manager`) gate, which is the WooCommerce-active check the context requires. If the role exists and `wp user get compat_shop_manager` fails, `wp user create compat_shop_manager compat-shop@example.com --role=shop_manager --user_pass=password`. If the role does NOT exist, echo a clear warning (e.g. "shop_manager role absent — WooCommerce not active yet") and do not fail the whole start.
3. The default admin already exists (wp-env creates admin/password); no action needed for it.

Express afterStart as a single `bash -c '...'` string (or a chain joined with `;`). Use `--user_pass=password` to match the project convention.

Create `tests/compat/VERSIONS.md`: a Markdown table with columns Plugin | Slug | Pinned Version | Pin Date | Source URL, one row per plugin (Pin Date = today, 2026-06-23). Add a one-line header noting these are the latest-stable releases as of the pin date and that Yoast SEO is the chosen SEO plugin (Rank Math not loaded). Keep VERSIONS.md next to the harness config (co-located with `.wp-env.json` so the record travels with the harness).
