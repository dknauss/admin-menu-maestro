<op_application_reference>
<!-- How to drive each Maestro op config-driven, against the maestro_config option. Sparse-diff shape from class-config.php. -->
- Natural baseline / reset between cases: `npx wp-env run cli wp option delete maestro_config`
- Rename `<slug>`:    `wp option update maestro_config '{"items":{"<slug>":{"title":"Renamed X"}}}' --format=json`
- Re-icon `<slug>` (top-level only): `'{"items":{"<slug>":{"icon":"dashicons-star-filled"}}}'`
- Hide `<slug>` from a role: `'{"items":{"<slug>":{"hidden_roles":["editor"]}}}'` (test each of editor / shop_manager; admin should still see it)
- Reorder top-level: `'{"top_order":["<slugA>","<slugB>",...]}'`  (drives custom_menu_order + menu_order)
- Reorder submenu:   `'{"sub_order":{"<parent_slug>":["<child_slug>","<child_slug>",...]}}'`
- After each op, RE-DUMP $menu/$submenu (same command as the Method header) and compare to the natural baseline. Then reload (re-run the dump simulating a fresh request, and/or reload the UI) to test persistence.
- **EXCEPTION — top-level reorder:** the post-`admin_menu` `$menu` dump does NOT reflect top-level order. `Replay::replay()` (admin_menu @ PHP_INT_MAX) applies rename/icon/visibility/submenu-order only; top-level order is applied separately via the `custom_menu_order` + `menu_order` filters, which run when the menu is rendered (`includes/class-replay.php:58-60`). So classify top-level Reorder cells from the **effective rendered order** — read the admin sidebar order in the UI/DOM (Playwright `#adminmenu` `.menu-top` sequence), or in WP-CLI apply the filter explicitly: `wp eval 'global $menu; add_filter("custom_menu_order","__return_true"); $order = apply_filters("menu_order", wp_list_pluck($menu, 2)); var_export($order);'`. Submenu reorder, rename, icon, and hide ARE visible in the raw post-replay dump.
- Per-role: run the dump/observe as each user (e.g. `wp eval --user=compat_editor ...` or log in via UI at http://localhost:8890/wp-admin/). Hide MUST be checked as admin (still visible), compat_editor, and compat_shop_manager.
</op_application_reference>

<replay_behavior_reference>
<!-- Predicts likely degraded/broken cells; confirm at runtime, do not assume. From class-replay.php / class-ordering.php. -->
- Maestro replay runs at admin_menu PHP_INT_MAX (after Woo). Rename/icon/hide mutate the globals THEN; if Woo re-registers or injects LATER than PHP_INT_MAX (rare) the override could be missed — confirm.
- Rename matches by slug (`$row[2]`). If Woo's title carries a count badge span, Maestro overwrites title index [0] wholesale → badge likely LOST on rename (degraded candidate). Confirm via dump.
- Reorder top-level uses custom_menu_order/menu_order with Ordering::top() resilience (named-first, newcomers appended, orphans skipped). Woo conditional/late top-level injection = newcomer → appended, may not honor desired slot (degraded candidate). Confirm.
- Submenu reorder via Ordering::submenu() reorders surviving rows by slug.
- Hide is cosmetic unset() per-role; page still loads by direct URL (NOT an access break → degraded/safe, not broken, unless something else breaks). Confirm 403-vs-loads by hitting the page URL as the hidden role.
- Re-icon is top-level only ($menu index 6). Submenu re-icon is N/A → classify with rationale per SCHEMA note.
</replay_behavior_reference>
