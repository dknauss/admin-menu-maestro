## Implementation Decisions (LOCKED by the code review)

### HARD-01 — scope `custom_menu_order`
- The bug: [`class-replay.php:59`](../../../includes/class-replay.php#L59) registers `add_filter( 'custom_menu_order', '__return_true' )` **unconditionally** in the constructor, claiming core's menu-order machinery on every admin load even with no stored `top_order`.
- **Fix direction:** enable the machinery only when a non-empty `top_order` exists in config. Preferred shape: replace `'__return_true'` with a small instance predicate (e.g. `has_top_order()`) used as the `custom_menu_order` callback, so it returns `true` only when `! empty( $cfg['top_order'] )` and `false` otherwise. `reorder_top()` ([`class-replay.php:159`](../../../includes/class-replay.php#L159)) already no-ops on an empty `top_order` via `Ordering::top()`, so this is a defense-in-depth alignment, not a behavior change for users who *have* set an order.
- **Test (red-first):** unit-test the gating predicate — returns `false` for empty/absent `top_order`, `true` for a non-empty one. `Ordering::top()` already carries coverage (OrderingTest). Add an integration assertion that the `custom_menu_order` filter is not forced `true` when config has no `top_order`.

### HARD-02 — bound config payload
- The gap: [`class-config.php`](../../../includes/class-config.php) `sanitize()` caps nothing. Unbounded title length, unbounded `items` / `top_order` / `sub_order` counts, unbounded `hidden_roles` length, and a data-URI icon stored **verbatim** ([`class-config.php:246`](../../../includes/class-config.php#L246)). `save()` uses `update_option( …, false )` so the option is **not autoloaded** — blast radius is limited, threat requires the `manage_options` capability; this is defense-in-depth against stored-option bloat, not a direct vuln.
- **Fix direction:** add deterministic caps inside the pure `sanitize()` path. Over-limit titles/data-URIs are **truncated**; over-limit collection entries are **dropped** (keep the first N, deterministic order). Caps live as named class constants so the tests and the code share one source of truth. The data-URI cap is measured on the raw string byte length and applied in the `'data'` branch of `sanitize_icon()` (drop to `''` if over — an over-large icon is invalid, not silently truncated mid-base64).
- **Test (red-first):** unit tests on `sanitize()` / `sanitize_icon()` — valid input under the caps passes through unchanged; over-limit title truncates to the cap; over-limit entry counts drop to the cap; an over-length data-URI is rejected to `''`. Pure logic, no WP needed for the count/length branches (the url/data branches already split unit vs integration today — keep that split).

### HARD-03 — save-race E2E coverage
- PR #36 hardened the single-flight save chain in [`assets/maestro.js`](../../../assets/maestro.js): `waitForSaveIdle()`, `cancelQueuedAutosave()`, Reset All awaiting in-flight saves and checking `response.ok`. No automated regression coverage locks it in.
- **Fix direction:** Playwright E2E that intercept/slow `POST /maestro/v1/config` (route handler with a delay) and assert three races:
  - **(a)** slow REST response + Exit → Exit waits for the in-flight/queued save to settle (not just the debounce timer) before leaving edit mode.
  - **(b)** pending rename (queued autosave) + Reset All → the queued autosave is cancelled and the DELETE wins; final stored state is reset, not the rename.
  - **(c)** in-flight save + Reset All → Reset All waits for the save to settle before issuing DELETE, and only reloads on a successful DELETE.
- **Test-only:** no production code change expected. If a race is found *not* to hold, that's a defect to fix in `maestro.js` (raise it; don't weaken the assertion). Follow the existing E2E patterns/selectors in `tests/e2e` (class-based selectors, the wave-boundary full-suite gate from Phase 9).

### Claude's Discretion (planner picks exact values; suggested defaults)
- **HARD-02 cap values** — internal sanity bounds, not user-facing product numbers. Suggested defaults the planner may adopt or adjust with a one-line rationale:
  - title max length: **200** chars (admin menu titles are short; generous headroom).
  - `items` map / `top_order` / `sub_order` parents: **200** entries each; children per `sub_order` parent: **200**.
  - `hidden_roles` per item: **50** (far above any real site's role count).
  - data-URI max length: **~128 KB** raw string (a 20×20 admin icon is tiny; this is generous but bounds abuse). Confirm against what a legitimately-encoded bundled icon needs.
- **HARD-01** — whether the predicate is a named private method vs an inline closure; whether to also short-circuit `reorder_top()` early (cosmetic; it already no-ops).
- **HARD-03** — whether to inject a tiny test fixture/mu-plugin to make REST timing deterministic, or rely on Playwright route delay alone.
- Plan granularity: whether HARD-01 + HARD-02 (both PHP, both `includes/`, both red-first unit) share one plan/wave and HARD-03 (JS/E2E) is a second, or all three split. Decide per dependency + TDD at plan time.
