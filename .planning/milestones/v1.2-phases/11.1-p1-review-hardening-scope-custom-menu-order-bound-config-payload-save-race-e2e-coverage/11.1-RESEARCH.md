## Architecture Patterns

### HARD-01: Predicate-gated filter

The predicate pattern mirrors `reorder_top()` exactly ‚Äî it reads from `$this->config->get()` and returns a boolean. The constructor registers it as the `custom_menu_order` callback instead of `'__return_true'`.

```php
// class-replay.php ‚Äî constructor (line 59 replacement)
add_filter( 'custom_menu_order', array( $this, 'has_top_order' ) );

// New private method (no WP dependencies ‚Äî pure config read):
private function has_top_order() {
    $cfg = $this->config->get();
    return ! empty( $cfg['top_order'] );
}
```

**Unit test seam:** `has_top_order()` is pure relative to a given Config state ‚Äî testable by constructing `Replay` with a stub `Config` that returns a fixed config array.

**Integration assertion:** After constructing `new Replay( new Config() )` with empty config, assert `has_filter( 'custom_menu_order' )` returns a non-false value (hook IS registered), but `apply_filters( 'custom_menu_order', false )` returns `false` (the predicate returns false, so machinery stays off). With a non-empty `top_order`, `apply_filters( 'custom_menu_order', false )` returns `true`.

NOTE: The predicate is registered on the filter, not just a one-time check ‚Äî WordPress calls the `custom_menu_order` filter every admin load, so the callback must read config at call time, not at constructor time. The `config->get()` call in `has_top_order()` is cached in-request by `Config::$cache`, so there is no per-request penalty.

### HARD-02: Caps as class constants + sanitize() enforcement

```php
// class-config.php ‚Äî add to the class body (before $cache):
const MAX_TITLE_LENGTH     = 200;
const MAX_ITEMS            = 200;
const MAX_ORDER_ENTRIES    = 200;
const MAX_SUB_ORDER_CHILDREN = 200;
const MAX_HIDDEN_ROLES     = 50;
const MAX_DATA_URI_BYTES   = 131072; // 128 KB

// In sanitize() ‚Äî title branch (after sanitize_text_field):
if ( mb_strlen( $entry['title'], '8bit' ) > self::MAX_TITLE_LENGTH ) {
    $entry['title'] = mb_substr( $entry['title'], 0, self::MAX_TITLE_LENGTH, '8bit' );
}

// In sanitize() ‚Äî items loop: cap total items
if ( count( $out['items'] ) >= self::MAX_ITEMS ) {
    break; // or continue ‚Äî deterministic: first N slugs win
}

// In sanitize() ‚Äî top_order after array_map:
$out['top_order'] = array_slice( $out['top_order'], 0, self::MAX_ORDER_ENTRIES );

// In sanitize() ‚Äî sub_order children:
$children = array_slice( $children_mapped, 0, self::MAX_SUB_ORDER_CHILDREN );

// In sanitize() ‚Äî hidden_roles (already array_intersect limited by valid roles):
if ( count( $roles ) > self::MAX_HIDDEN_ROLES ) {
    $roles = array_slice( $roles, 0, self::MAX_HIDDEN_ROLES );
}

// In sanitize_icon() ‚Äî 'data' branch (line 246 replacement):
case 'data':
    if ( strlen( $icon ) > self::MAX_DATA_URI_BYTES ) {
        return ''; // over-limit: drop, not truncate (mid-base64 is corrupt)
    }
    return $icon;
```

**Why `strlen` not `mb_strlen` for data-URI:** the data-URI is ASCII-only base64 ‚Äî byte length and character length are identical. `strlen()` is faster and appropriate.

**Why `mb_strlen( ..., '8bit' )` for title:** admin menu titles may contain multibyte characters; `8bit` mode measures raw byte length, which is what the database ultimately stores.

### HARD-03: Playwright route intercept pattern

```typescript
// Delay helper (mirrors existing POST_SAVE predicate pattern)
const delayPost = async ( page: Page, delayMs: number ) => {
    await page.route(
        url => url.includes( '/maestro/v1/config' ) && /* method check via request */,
        async ( route ) => {
            if ( route.request().method() === 'POST' ) {
                await new Promise( r => setTimeout( r, delayMs ) );
            }
            await route.continue();
        }
    );
};
```

The `page.route()` approach is sufficient without a mu-plugin; `route.continue()` after a delay lets the real server process the request and return a real response, so `response.ok` is the authentic server outcome. This avoids a mu-plugin dependency.

### Anti-Patterns to Avoid

- **Truncating data-URIs mid-base64:** a truncated base64 string is cryptographically corrupt and will fail to decode/render. The cap must drop the entire icon to `''`, not substr it.
- **Testing `custom_menu_order` by checking `has_filter` alone:** `has_filter( 'custom_menu_order' )` is always `true` after the constructor (the hook is registered). The assertion must call `apply_filters( 'custom_menu_order', false )` with the predicate in context to confirm the returned value.
- **Magic numbers in tests:** all cap values in test assertions MUST reference `Config::MAX_*` constants, not literals. This is mandated by CONTEXT.md and ensures tests break when caps change, not silently pass.
- **Using `mb_strlen` for data-URI cap:** data-URI is ASCII; `strlen()` is correct and 3‚Äì5x faster.

## Common Pitfalls

### Pitfall 1: `custom_menu_order` vs `menu_order` filter pairing

**What goes wrong:** Disabling/gating `custom_menu_order` without understanding that WordPress only calls the `menu_order` filter if `custom_menu_order` returns `true` for the current request. If `has_top_order()` returns `false`, the `menu_order` callback (`reorder_top`) is never called ‚Äî which is the desired behavior. BUT the `menu_order` filter registration at line 60 is unconditional and harmless (WordPress ignores it when `custom_menu_order` returns false).

**Why it happens:** The two filters are coupled: `custom_menu_order` is the gate; `menu_order` is the payload. Removing the `menu_order` hook registration is unnecessary; only the gate matters.

**How to avoid:** Keep both `add_filter` calls in the constructor. Only replace the `custom_menu_order` callback. The `reorder_top()` already handles empty `top_order` gracefully via `Ordering::top()`.

### Pitfall 2: `sanitize()` items loop ‚Äî iteration order under PHP

**What goes wrong:** PHP `foreach` on an associative array preserves insertion order. If the limit is applied by breaking after `count($out['items']) >= MAX_ITEMS`, the "first N" are the first N slugs as they appear in the incoming payload. This is deterministic only if the payload order is deterministic. The JS `buildConfig()` uses `Object.keys(model)` which in V8 iterates in insertion order for string keys ‚Äî deterministic across re-saves. This is fine; document it.

**How to avoid:** The `break` strategy is correct. Document in code that "first N" means insertion order of the incoming JSON object.

### Pitfall 3: Playwright route delay and DELETE timing

**What goes wrong:** In race (c) (in-flight save + Reset All), after the delay the POST lands and the server responds, then the DELETE fires. If the test `waitForResponse` is set up too early (before the route delay is installed), or catches the wrong response, the assertion may succeed vacuously.

**How to avoid:** Set up route interception BEFORE navigating to the page. Use `page.waitForResponse()` with a method check for `DELETE` in race (b)/(c), not just the URL. Assert `response.ok()` on the DELETE response directly.

### Pitfall 4: Unit-testing `has_top_order()` if it's private

**What goes wrong:** Private methods cannot be called directly in PHPUnit. The test must drive the predicate through the filter mechanism: register the hook, then call `apply_filters('custom_menu_order', false)`.

**How to avoid:** The integration test for HARD-01 calls `apply_filters()` on the WP hook; the unit test stubs Config and asserts the filter return value. A named private method is testable this way ‚Äî no reflection hacks needed.

### Pitfall 5: E2E - race (a) Exit href check

**What goes wrong:** `onExit()` calls `window.location.href = D.exitUrl`. Playwright's `page.waitForURL()` may resolve immediately if the URL was already the exit URL, or may race if navigation is fast. The assertion must wait for the save to settle FIRST, then detect navigation.

**How to avoid:** Use `Promise.race([page.waitForURL(D.exitUrl), page.waitForResponse(POST_SAVE)])` is NOT correct. Instead: intercept POST with a delay, click Exit, assert the page does NOT navigate immediately (still on edit URL), then let the delayed POST resolve and assert eventual navigation to the exit URL.

### Verified: `doResetAll()` race code (maestro.js:1060‚Äì1085)

```javascript
function doResetAll( e ) {
    e.preventDefault();
    if ( ! window.confirm( I.confirmAll ) ) { return; }
    var button = e.currentTarget;
    if ( button ) { button.disabled = true; }
    setStatus( 'saving' );
    cancelQueuedAutosave();          // kills saveTimer + savePending

    ( inFlight || Promise.resolve() )   // awaits any in-flight POST
        .then( function () {
            return fetch( D.restUrl, { method: 'DELETE', ... } );
        } )
        .then( function ( r ) {
            if ( ! r.ok ) { throw new Error( 'HTTP ' + r.status ); }
            window.location.reload();
        } )
        .catch( function () {
            if ( button ) { button.disabled = false; }
            setStatus( 'error' );
        } );
}
```

Races (b) and (c) both flow through this code. The E2E must assert: DELETE does NOT fire while `inFlight` is still pending; after the POST settles, the DELETE fires; if DELETE succeeds, page reloads.

### Verified: `onExit()` code (maestro.js:1087‚Äì1095)

```javascript
function onExit( e ) {
    if ( saveTimer || inFlight ) {
        e.preventDefault();
        waitForSaveIdle().then( function () {
            window.location.href = D.exitUrl;
        } );
    }
}
```

Race (a): if `saveTimer` is set (debounced, not yet fired) OR `inFlight` is set (POST in flight), navigation is prevented and deferred until `waitForSaveIdle()` resolves. E2E must verify the page stays on the edit URL while the delayed POST is in-flight, then navigates after.

## Open Questions

1. **HARD-01 predicate visibility**
   - What we know: `has_top_order()` is private and called as a method callback.
   - What's unclear: whether to expose it as `public` for testability via `apply_filters` (preferred) vs using a reflection approach.
   - Recommendation: Keep private; test via `apply_filters('custom_menu_order', false)` in integration. Unit-test by setting up a lightweight `Replay` instance with a stub or `Config` backed by WP option.

2. **HARD-02 `mb_strlen` vs `strlen` for title cap**
   - What we know: `sanitize_text_field()` runs first and strips non-printable characters; the resulting title is UTF-8.
   - What's unclear: whether the cap is in characters or bytes ‚Äî both defensible, but the constant name should be explicit.
   - Recommendation: Cap in bytes (`strlen`) for storage safety; name the constant `MAX_TITLE_BYTES` to make the semantics clear. 200 bytes generously covers ~66 CJK characters or 200 ASCII characters.

3. **HARD-03 race (a) ‚Äî exit URL detection**
   - What we know: `D.exitUrl` is set server-side and is typically the admin dashboard or referring page. Tests target `localhost:8889`.
   - What's unclear: whether `page.waitForURL()` with a regex or partial match is more robust than waiting for navigation.
   - Recommendation: Assert that while the delayed POST is in-flight, the page URL still contains `maestro_edit=1`; after POST resolves, assert URL no longer contains `maestro_edit=1` (or matches `D.exitUrl`). Avoids hardcoding the exact exit URL.

**Route interception approach (confirmed no mu-plugin needed):**

```typescript
// Delay pattern for races (a) and (c):
await page.route(
    url => url.pathname.includes( '/maestro/v1/config' ),
    async ( route ) => {
        if ( route.request().method() === 'POST' ) {

        }

    }
);
```

`route.continue()` after the delay lets the actual WordPress server process the request and return a real HTTP response ‚Äî `r.ok` reflects the server's actual status, not a mocked one. This is the simplest, most accurate approach; no mu-plugin required.
