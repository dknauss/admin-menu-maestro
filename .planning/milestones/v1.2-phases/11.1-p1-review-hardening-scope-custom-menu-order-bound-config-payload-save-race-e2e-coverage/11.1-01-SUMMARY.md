## Decisions Made
- `has_top_order()` is public (not private): WP's `call_user_func_array()` cannot access private methods registered as filter callbacks — PHP raises a TypeError at hook dispatch. Public visibility is required. The method is still not part of the plugin's documented API.
- Read config at call time: WP invokes `custom_menu_order` on every admin load; reading config in the callback (not in the constructor) is correct because Config caches in-request anyway.
- Integration seam preferred over unit: unit bootstrap does not load class-replay.php; testing via `apply_filters('custom_menu_order', false)` through the live WP filter is the lower-friction, fully-sufficient seam per the plan's tdd_note.

**1. [Rule 1 - Bug] Changed has_top_order() visibility from private to public**
- **Found during:** Task 1 (GREEN run of integration tests)
- **Issue:** PHP raises `TypeError: cannot access private method Maestro\Replay::has_top_order()` when WordPress's `call_user_func_array()` attempts to invoke the callback registered via `add_filter()`. Private methods cannot be called from outside the class, even by WP's hook dispatcher.
- **Fix:** Changed `private function has_top_order()` to `public function has_top_order()`. This is the minimum change required — WP hook dispatch requires at least public visibility for `array($this, 'method')` callbacks.
