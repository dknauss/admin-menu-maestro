<objective>
Make changed state legible (UX-01): every item differing from its default shows a
visible, non-color-only "modified" indicator in edit mode, and per-item reset is a
discoverable, keyboard-reachable affordance that clears both the delta and the
indicator. Update public docs to retire the v1 keyboard-reorder limitation.

Purpose: Users currently cannot tell which items they have changed, and per-item
reset (which already exists as `.maestro-reset-item` in the shared panel) is not
clearly surfaced as the discoverable affordance UX-01 requires. This plan adds the
indicator and hardens the reset discoverability, driven by the unit-tested diffItem
from Plan 01.

Output: A modified indicator on each changed row (glyph/badge + accessible text, AA
contrast), a clearly-labelled keyboard-reachable per-item reset, live refresh on
every mutation, e2e coverage, and doc updates.
</objective>

<indicator_decision>
DECISION (Claude's Discretion #3): **Non-color-only indicator = a small filled dot/
asterisk glyph on the row PLUS `.screen-reader-text` "(modified)"**, with an
accessible name update on the row. Color is supplementary, never the sole signal
(WCAG 1.4.1). Contrast of the glyph against the admin-menu background must meet WCAG
1.4.11 (>= 3:1 for the graphical object) — the screen-reader-text carries the semantic
meaning so color is never the only signal. The per-item reset affordance is the
existing `.maestro-reset-item` button in the selected item's panel — already
keyboard-reachable; this plan ensures it is clearly labelled, focus-visible, and only
enabled/emphasized when the item is actually modified, making it discoverable in
context rather than hidden.
</indicator_decision>

  <name>Task 1: Modified indicator — render + live refresh via diffItem, harden reset discoverability</name>
  <files>assets/maestro.js, assets/maestro.css, includes/class-assets.php</files>
  <action>
    PREREQUISITE — add `.screen-reader-text` CSS (research correction #4): `.screen-reader-text`
    is NOT currently present in assets/maestro.css (verified). Before/with the badge work,
    add the canonical WordPress-core clip-path implementation to assets/maestro.css so the
    "(modified)" text is hidden from sighted users but exposed to AT (do NOT use
    display:none or visibility:hidden — those also hide it from screen readers):
    ```
    .maestro-editing .screen-reader-text {
        border: 0;
        clip-path: inset(50%);
        height: 1px;
        margin: -1px;
        overflow: hidden;
        padding: 0;
        position: absolute;
        width: 1px;
        word-wrap: normal !important;
    }
    ```
    Scoping under `.maestro-editing` keeps it confined to edit mode and avoids colliding
    with any WP core `.screen-reader-text` already on the page.

    Add refreshModifiedIndicator(slug) to assets/maestro.js. For the given slug:
    - def = model[slug].isSub ? pristineSub(slug) : pristineTop(slug).
    - result = window.maestroLogic.diffItem(model[slug], def).
    - li = liForSlug(slug). Toggle class 'maestro-modified' on the row based on
      result.modified. When modified, ensure a non-color indicator node exists inside
      the row (e.g. a <span class="maestro-modified-badge" aria-hidden="true">•</span>
      glyph PLUS a sibling <span class="screen-reader-text"> with the localized
      "(modified)" text); when not modified, remove them. Do NOT rely on color alone.
    - Update the row's accessible context (e.g. append the screen-reader-text) so AT
      users perceive the modified state.

    Call refreshModifiedIndicator(slug) after EVERY mutation point so the indicator is
    always live:
    - commitRename() (after m.title change)
    - icon picker choose() (after m.icon change)
    - visibility checkbox change handler (after hiddenRoles change)
    - resetSelected() (must clear it)
    - the keyboard/drag reorder paths do NOT change modified-state (order isn't a
      per-item field), so they need no indicator refresh.
    On init(), after the model is built, run refreshModifiedIndicator for every slug
    so pre-existing (already-saved) overrides show their indicator on page load.

    Harden the discoverable per-item reset (the existing panel `.maestro-reset-item`
    button, panel.resetBtn): give it a clear localized label (already I.resetItem),
    ensure it is keyboard-focusable (it is a <button> — confirm no tabindex=-1), add a
    focus-visible style, and reflect the selected item's modified state by
    enabling/emphasizing it only when diffItem reports modified (e.g. toggle a
    'is-modified' class / disabled attribute in populatePanel via diffItem). It must
    remain discoverable without documentation: visible in the panel whenever an item
    is selected.

    i18n: add a localized "(modified)" string (e.g. `modified`) to the i18n map in
    includes/class-assets.php via WP translation helpers + text domain; read via I.* .

    CSS (assets/maestro.css): style `.maestro-modified` / `.maestro-modified-badge`
    as a glyph/badge that is perceivable without color (shape + the screen-reader
    text). The badge glyph color MUST meet >= 3:1 contrast against the dark admin-menu
    background `#1d2327` (WCAG 1.4.11 graphical object) — e.g. amber `#dba617` or white
    `#fff` on `#1d2327` clears 3:1 easily; do NOT use a low-contrast grey. Add a
    focus-visible outline for `.maestro-reset-item`.

    Pure modified-state logic lives in maestroLogic.diffItem (unit-tested in Plan 01);
    this task is DOM/CSS glue covered by e2e in Task 2 — no new unit test here.
  </action>
