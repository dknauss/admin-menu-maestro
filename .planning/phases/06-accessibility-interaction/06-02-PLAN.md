<objective>
Close the v1 keyboard-reordering gap (A11Y-06): make menu items reorderable up/down
with the keyboard alone, announcing each move to assistive technology, reusing the
pure reorderMove helper from Plan 01 and the existing debounced autosave.

Purpose: v1.0 shipped with a documented "keyboard reorder unsupported" limitation.
This plan removes it without new architecture — the keyboard path produces the exact
same top_order/sub_order payload the mouse drag already produces.

Output: An Alt+Arrow keyboard reorder interaction on the selected item, explicit
focus restoration after each move so presses chain, AT move announcements via
wp.a11y.speak() (polite for moves, assertive for boundary clamps), and a
keyboard-only e2e walkthrough proving the new order persists across reload.
</objective>

<keyboard_model_decision>
DECISION (Claude's Discretion #1): **Modifier+arrow — Alt+ArrowUp / Alt+ArrowDown on
the selected item.** Justification (matches CONTEXT.md's recommended default):

- Simplest and most discoverable; native to WP admin reorder conventions.
- Lowest regression risk: it layers on the EXISTING click/Enter/Space selection model
  (`bindMenuSelection`) without disturbing the jQuery-UI sortable drag layer.
- Alt is chosen over Ctrl/Cmd to avoid clobbering browser/OS shortcuts and the
  rename input. The handler only fires when a maestro item row (not a form control or
  popover) has focus/selection, so it cannot hijack typing in the rename field.
- ARIA grab/drop (aria-grabbed) is the acceptable alternative per A11Y-06 but is
  heavier, less discoverable, and is being deprecated in ARIA — modifier+arrow is the
  better fit here. We still expose intent via aria-keyshortcuts on the selected item.
</keyboard_model_decision>

<task type="auto">
  <name>Task 1: Keyboard reorder handler — Alt+Arrow moves the selected item, restores focus, announces, autosaves</name>
  <files>assets/maestro.js, assets/maestro.css, includes/class-assets.php</files>
  <action>
    Add a keydown handler (extend the existing `bindMenuSelection` listener on
    #adminmenu, or add a sibling listener) that fires only when:
    - e.altKey is true and e.key is 'ArrowUp' or 'ArrowDown', AND
    - a maestro item is currently selected (selectedSlug set) and focus is NOT inside
      a popover or a form control (guard: e.target.closest('.maestro-popover, input,
      button') returns null, OR the selected row itself is the active context).

    On a valid Alt+Arrow:
    1. e.preventDefault() (stop the page from scrolling).
    2. Determine the list scope from model[selectedSlug].isSub:
       - top-level: read current order from
         `#adminmenu > li.menu-top.maestro-item[data-maestro-slug]` -> slugs.
       - submenu: read siblings from the parent's
         `.wp-submenu > li.maestro-subitem[data-maestro-slug]` -> slugs.
    3. dir = e.key === 'ArrowUp' ? 'up' : 'down'.
    4. newOrder = window.maestroLogic.reorderMove(currentSlugs, selectedSlug, dir).
    5. If newOrder is unchanged (boundary clamp): announce a boundary message
       ASSERTIVELY (a failed action / error-type feedback) via the speak() helper —
       see the speak()-politeness step below — using I.moveAtTop / I.moveAtBottom
       (e.g. "Posts is already first"), then return — do NOT autosave (nothing changed).
    6. Else: physically reorder the <li> nodes in the DOM to match newOrder
       (re-append in order under the correct parent ul). Then explicitly restore
       focus to the moved row's anchor — REQUIRED, because re-appending a node
       detaches it and the browser drops focus to <body>:
       ```
       var movedLi = liForSlug( selectedSlug );
       if ( movedLi ) {
           var focusTarget = movedLi.querySelector( 'a' ) || movedLi;
           focusTarget.focus( { preventScroll: true } );
       }
       ```
       This focus({ preventScroll: true }) call MUST run AFTER the re-append, not
       before, so a subsequent Alt+Arrow chains off the still-focused item instead of
       starting over. Then announce a POLITE move message with the new position, e.g.
       I.moved (sprintf-style) -> "Posts moved down, position 3 of 8". Then call
       scheduleAutosave() — this reuses the exact debounced full-config POST the drag
       `stop` handler uses, so buildConfig() reads the new DOM order.
    7. Set aria-keyshortcuts="Alt+ArrowUp Alt+ArrowDown" on selected rows (discovery
       hint for AT); add a `.screen-reader-text` hint is optional.

    speak() politeness (research correction #3): the existing speak() helper
    (maestro.js lines 59-63) currently calls `window.wp.a11y.speak( message )` with NO
    second argument, so every announcement is polite. wp.a11y.speak() DOES accept a
    second argument: `speak( message, 'assertive' )`. Make the MINIMAL change to pass
    the politeness level through: give speak() an optional second parameter, e.g.
    `function speak( message, politeness ) { ... window.wp.a11y.speak( message,
    politeness ); }` (wp.a11y.speak treats an undefined/omitted level as polite, so
    existing single-arg callers are unaffected). Then:
    - Successful move announcements: call speak( msg ) — stays POLITE (does not
      interrupt the screen reader mid-sentence).
    - Boundary-clamp announcements: call speak( msg, 'assertive' ) — the failed action
      needs immediate acknowledgment.
    Do NOT build a custom live region; wp.a11y.speak() already clears-before-set (Safari
    VoiceOver repeated-string workaround).

    Localized strings: add to the i18n map in includes/class-assets.php (and read via
    I.* in maestro.js): `moved`, `moveAtTop`, `moveAtBottom`. Use WP translation
    helpers (esc_html__/__) with the maestro-menu-editor text domain. For the
    position-aware message, format in JS (e.g. replace %1$s/%2$d/%3$d placeholders)
    to keep it translatable.

    CSS (assets/maestro.css): add a brief visual focus/move affordance if needed
    (e.g. ensure .maestro-selected has a clear focus ring) — styling only, no logic.

    Pure logic stays in maestroLogic.reorderMove (already unit-tested in Plan 01);
    this task is DOM glue + announcements, covered by e2e in Task 2 — so no new
    unit test here.
