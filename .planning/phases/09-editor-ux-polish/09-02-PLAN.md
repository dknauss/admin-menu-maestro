<objective>
UX-03 status redesign: replace the verbose idle status with a short, glanceable
"Edit Mode" indicator that signals mode by dashicon + text + colour (not colour
alone, WCAG 1.4.1), and split the single status element into a PERSISTENT mode
label plus a SEPARATE transient save-status element.

Purpose: Today one `statusEl` alternates between "Editor active — click an item to
edit." and Saving…/Saved/Save failed — so the persistent mode signal is destroyed
every time a save fires, and the idle copy is verbose. Splitting them keeps "Edit
Mode" always legible and lets save feedback come and go in its own element.

Output: i18n change (idle → "Edit Mode") + new modeLabel key (with the LocalizationTest
update in the same commit), the buildToolbar/setStatus DOM split routed through the
Plan-01 modeStatusLabel seam, mode-label CSS, and e2e assertions.

RECONCILIATION (for the plan-checker): roadmap success criterion #1 literally reads
"Menu Edit Mode". The user LOCKED the shorter "Edit Mode" (09-CONTEXT.md). This is a
deliberate wording refinement that satisfies the INTENT of criterion #1 (short,
glanceable, non-colour-signalled mode indicator) — same reconciliation pattern Phase
8 used for REL-06. Treat "Edit Mode" as the locked target, NOT a missed criterion.
</objective>
