<objective>
Switch the bundled icon picker from thin outline Bootstrap Icons to solid `*-fill`
variants so the glyphs read at a weight that mixes with WordPress's solid
dashicons (ICON-01). The icon-bundle generation is a data transformation, so it is
built test-first: a pure `resolveIcon(name)` policy is unit-tested before the
bundle is regenerated.

Purpose: The current 87-icon bundle is outline-weight and reads thin/light next to
dashicons, so the two picker tabs don't mix. 670 `*-fill.svg` variants are
available in the same MIT dependency; 58 of the 87 curated names have a direct
`-fill`, and the remaining 29 need an explicit per-name fallback (solid synonym or
retained outline) — never a silent drop.

Output: A pure, exported `resolveIcon` in the generator with node:test coverage; a
regenerated `includes/icons-bootstrap.php` (still 87 entries, now solid where
possible) that ships via `assets/`→`includes/` build copy; and a confirmed
edit-mode payload-budget contract.
</objective>

<fill_resolution_decision>
DECISION (Claude's Discretion #1, resolved here — this is the one opus judgment
call in the phase): per CURATED name, resolve in this priority order, and NEVER
drop an icon:

1. **Prefer `<name>-fill`** when `node_modules/bootstrap-icons/icons/<name>-fill.svg`
   exists (58 of 87 names). This is the ICON-01 happy path.
2. **Else apply an explicit SYNONYM_FILL map** for names whose meaning survives a
   solid substitute (verified present in node_modules). Confirmed candidates:
   `files`→`file-earmark-fill`, `journal-text`→`journal-fill`,
   `pencil-square`→`pencil-fill`, `images`→`image-fill`, `folder2-open`→`folder-fill`,
   `cart3`→`cart-fill`, `person-circle`→`person-fill`, `person-lock`→`person-fill-lock`
   (verify; else `person-fill`), `graph-up`→`graph-up-arrow` only if a fill exists
   (else outline). Each mapping MUST be validated by existsSync in the generator and
   asserted in the test; a mapping whose target is missing is a hard error, not a
   silent skip.
3. **Else retain the outline `<name>`** and record it in a `RETAINED_OUTLINE`
   list printed by the generator. Names with no sensible solid form — e.g.
   `speedometer2`, `sliders`, `tools`, `wrench`, `list-ul`, `table`, `code-slash`,
   `link-45deg`, `card-text`, `layout-text-window`, `music-note-beamed`,
   `gear-wide-connected`, `server`, `globe`, `receipt`, `truck`, `cash-stack`,
   `shop`, `clipboard-data`-adjacent — stay outline. These are the minority and read
   acceptably; the side-by-side review in Plan 03 confirms whether any still read too
   light (Heroicons fallback is deferred unless that review fails — Claude's
   Discretion #2, NOT pursued in this plan).

The policy is encoded as the tested behavior of `resolveIcon`. The invariant the
test enforces: `resolved.length === CURATED.length` (87, no drops) and every
resolved source file exists.
</fill_resolution_decision>

The current per-name transform to preserve (baking menu-grey, stripping size/class):
```js
svg = svg.replace(/\s(width|height|class)="[^"]*"/g, '')
         .replace(/fill="currentColor"/g, `fill="${MENU_GREY}"`)
         .replace(/>\s+</g, '><').trim();
if (!/fill="/.test(svg)) svg = svg.replace('<svg ', `<svg fill="${MENU_GREY}" `);
return 'data:image/svg+xml;base64,' + Buffer.from(svg,'utf8').toString('base64');
```
