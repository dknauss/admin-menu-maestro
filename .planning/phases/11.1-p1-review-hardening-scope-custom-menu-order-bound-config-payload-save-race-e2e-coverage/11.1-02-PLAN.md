<objective>
HARD-02 — bound the config payload size in `Config::sanitize()`. Today
`includes/class-config.php` sanitises shape and content well but caps NOTHING (no
strlen/count/MAX anywhere). An authenticated editor (manage_options) can store an
unbounded `maestro_config` option: arbitrarily long titles, unbounded items/top_order/
sub_order/hidden_roles, and a data-URI icon stored verbatim. Add deterministic caps as
named class constants enforced inline in the pure sanitize() path.

Purpose: defense-in-depth against stored-option bloat. `save()` uses
`update_option( …, false )` (not autoloaded), so blast radius is limited and this is not
a direct vuln given the capability gate — but bounding the payload is correct hardening.

Output: six MAX_* constants on Config, cap enforcement in sanitize() + the sanitize_icon()
'data' branch, and red-first unit tests in tests/unit/ConfigSanitizeTest.php covering
every cap. Pure-PHP, no Docker: `vendor/bin/phpunit -c phpunit-unit.xml.dist` (the unit
bootstrap already loads class-config.php).
</objective>

CAP VALUES (Claude's Discretion — RESOLVED with rationale; defined as Config constants so
tests and code share one source of truth):
  MAX_TITLE_BYTES        = 200   // admin titles are short; 200 bytes ≈ 200 ASCII / ~66 CJK chars. Byte-measured (strlen) for storage-truthful semantics (research Open Q#2).
  MAX_ITEMS              = 200   // generous headroom over any real admin menu's item count.
  MAX_ORDER_ENTRIES      = 200   // top_order entries; matches MAX_ITEMS scale.
  MAX_SUB_ORDER_CHILDREN = 200   // children per sub_order parent.
  MAX_HIDDEN_ROLES       = 50    // far above any real site's role count (also already array_intersect-limited to valid roles).
  MAX_DATA_URI_BYTES     = 131072 // 128 KB raw string; ~57x the largest bundled Bootstrap icon (2,242 bytes, measured). Generous sanity bound, zero false positives on real icons.
</tdd_note>
